# Boronix v0.3.0 - Developer Experience Core

Boronix v0.3.0 introduces the **Developer Experience Core** phase, focusing on making Boronix application development easier to debug, inspect, and typecheck.

## Core Additions

### 1. Dev Error Page
When running `boronix dev`, any runtime loader, action, template rendering, API, or configuration error outputs a visually styled dark-mode error page.
- **Error Title & Message**
- **Match Pattern & Method**
- **Error Phase**: e.g., `page-loader`, `page-render`, `action`, `layout`.
- **Code Frame**: Displays the exact lines surrounding the error, with a caret pointing to the column location.
- **Clean Stack Trace**: Hides framework internal stack frames, highlighting project files.

### 2. Error & Not-Found Boundaries
We support nested file routing convention for custom error boundaries:
- `app/not-found.html` and `app/routes/<route>/not-found.html`
- `app/error.html` and `app/routes/<route>/error.html`

**Resolution Logic:**
- Looks for the closest boundary file to the matched route directory, cascading up to the global boundary under `app/`.
- If no custom HTML is found, cascades to the beautiful default Boronix page (for dev) or clean Generic page (for production).
- Calling `notFound()` inside API routes returns clean 404 JSON, while action routes properly report target mismatches.

### 3. CLI inspect Diagnostics
The `boronix inspect` command has been upgraded to inspect specific methods, actions, and output clean JSON:
- `boronix inspect /login?/login`
- `boronix inspect --method POST "/login?/login"`
- `boronix inspect /login --json` (outputs parseable JSON with exit code validation)

### 4. Typegen Polish
The `boronix typegen` command generates `.boronix/types/routes.d.ts` which includes:
- `BoronixRoute`: Union of all scanned routes (e.g. `/` | `/login` | `/exercises/[id]`).
- `BoronixRouteParams`: Parameter types for routes.
- `RouteParams<T>`: Utility type mapper.
